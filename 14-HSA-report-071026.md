## 代理型 AI (Agentic AI) 在醫療器材管理中的應用：衛生科學局 (HSA) 審查官內部實務指南
衛生科學局 (HSA) 審查官利用代理型 AI (Agentic AI) 工具能將醫療器材的上市前審查、上市後監管及法規合規流程自動化，大幅提升 80% 以上的行政效率並降低人工審查盲點。 本指南專為 HSA 內部法規審查官、技術專家及風險管理人員設計，全面解析如何將具備自主規劃、工具調用與反思能力的「代理型 AI 系列工具」深度融入醫療器材生命週期管理 (MDLM) 的核心日常工作。 [1] 
------------------------------
## 🧭 核心概念與系統架構## 什麼是代理型 AI (Agentic AI)？
傳統的 AI 生成工具（如標準大型語言模型）僅能依據單一提示詞 (Prompt) 進行單次、被動的文字輸出。
代理型 AI (Agentic AI) 則具備自主性 (Autonomy)、目標導向 (Goal-Oriented)、工具調用 (Tool Use) 以及反思修正 (Reflection) 能力。
HSA 審查官只需給定高階任務（例如：「評估此款 AI 診斷軟體的網路安全風險」），AI 代理便能自行拆解步驟、調用內部法規資料庫、分析廠商提交的技術文件 (CSDT)、生成追溯表，並在發現數據矛盾時自動啟動覆核機制。
## HSA 內部核心架構

## 🛡️ 模組一：上市前審查 (Pre-Market Review) 自動化
上市前審查是 HSA 審查官工作量最龐大的部分。利用 Agentic AI，審查官可以將傳統需要數週的「通用醫療器材技術文件 (CSDT)」比對工作縮短至數小時。
## 1. 廠商文件完整性與合規性自動查核 (Completeness Check)
當醫療器材製造商透過新加坡 MEDICS 系統 提交申請時，文件自動化代理 (Doc-Agent) 會立即啟動：

* 格式與結構掃描：依據 HSA GN-15 指南，自動解析 PDF 檔案，確認是否完整包含：器材描述、設計驗證、風險分析、製造工藝及臨床數據。
* 缺失交叉比對：若文件提及「生物相容性測試遵循 ISO 10993-1」，但附錄中缺乏相應的實驗報告，Agent 會在 3 秒內標記該漏洞，並自動草擬「補充資料通知書 (RFI)」。

## 2. 基於 GN-17 的風險分級驗證
審查官常遇到廠商「故意低報風險評級（例如將 Class C 申報為 Class B）」以加速取證。

* 自主推理鏈 (Chain-of-Thought)：法規代理 (Reg-Agent) 會調取器材的預期用途 (Intended Use) 與適應症 (Indications for Use)。
* 範例分析：
* 輸入：「此器材為用於心血管內科的導管，表面塗有抗凝血肝素。」
   * AI 推理：
   1. 檢索 Health Products (Medical Devices) Regulations。
      2. 識別關鍵詞「心血管系統」與「藥物結合器材 (Drug-Device Combination)」。
      3. 根據 Rule 13 (含有藥物成分之器材) 與 Rule 1 (與中樞循環系統直接接觸之侵入性器材)。
      4. 判定正確分級應為 Class D。
   * 輸出：若廠商申報 Class C，Agent 會立刻警示審查官，並列出違反的法規條款對照表。

## 3. 技術文件 (CSDT) 實質審查輔助
對於高風險的 Class C 與 Class D 器材，技術審核代理 (Tech-Agent) 會深入審查驗證與確認 (V&V) 報告：

* 生物相容性審查：提取細胞毒性、致敏性、皮內反應等實驗數據，自動與標準數據（如 ISO 10993 標準限值）進行比對，標記不合規的稀釋比例或對照組異常。
* 滅菌驗證：審查滅菌保證水平（SAL = 10⁻⁶）是否符合 ISO 11135（環氧乙烷滅菌）或 ISO 11137（輻射滅菌）參數，防止廠商用非標準的計算模型矇混過關。

## 💻 模組二：軟體即醫療器材 (SaMD) 與人工智慧器材專項審查
隨著 AI 醫療軟體（如 AI 肺結節篩檢、心電圖自動分析軟體）激增，HSA 審查官面臨著技術黑箱的挑戰。Agentic AI 內部工具能在此提供深度的黑箱解構。
## 1. 網路安全 (Cybersecurity) 自主審查流程
依據 HSA 頒布的《醫療器材網路安全指南》，網路安全專門代理 (Cyber-Agent) 會對廠商的軟體配置管理進行以下審查：

* 軟體物料清單 (SBOM) 深度解析：自動提取廠商申報的開源組件（如 Python 庫、OpenSSL 版本），聯網調取美國 [NIST NVD (國家漏洞資料庫)](https://nvd.nist.gov/)。
* 漏洞交叉關聯：若發現軟體包含已被披露的 CVE 高危漏洞（如 CVSS ≥ 8.0），且廠商未在風險緩解計畫中提及，Agent 將直接阻斷審查流程，將此列為「重大安全缺陷」。

## 2. 演算法偏誤 (Algorithm Bias) 與臨床驗證審查
臨床評估代理 (Clin-Agent) 專門負責拆解機器學習模型的訓練與測試數據結構：

| 審查維度 | 廠商提交常見漏洞 | Agentic AI 自主檢查機制 |
|---|---|---|
| 數據集代表性 | 僅使用單一西方國家數據集。 | 自動提取人口統計學數據（年齡、性別、種族），比對新加坡本地（華人、馬來人、印度人）人口結構，評估泛化風險。 |
| 黃金標準對比 | 採用模糊的 ground truth。 | 分析標註者的資歷與共識機制（例如：是否由 3 位資深放射科醫生雙盲審查），評估標註質量。 |
| 過度擬合風險 | 訓練集與驗證集數據交叉污染。 | 追溯數據拆分邏輯（Train/Val/Test Split），檢查是否有同一患者的多張影像同時存在於訓練與測試集。 |

## 🚨 模組三：上市後監管與主動式風險預警 (Post-Market Surveillance)
器材上市不代表風險結束。Agentic AI 在上市後監管 (PMS) 中扮演「全天候哨兵」的角色。 [2] 
## 1. 不良事件 (Adverse Events) 的多源情報自動搜集
傳統上，HSA 依賴廠商主動通報或醫院提交報告。現在，情報搜集代理 (Intel-Agent) 能做到主動式、跨國界的風險追蹤：

* 跨國法規機關連動：每日自動爬取並結構化分析美國 FDA MAUDE 資料庫、澳洲 TGA、歐盟 EUDAMED 的召回與警告公告。
* 社群與文獻監測：監測醫學論壇、學術文獻 (PubMed) 以及社交媒體上醫生對特定型號人工關節、心臟支架的抱怨（如「早期鬆動」、「塗層脫落」）。

## 2. 不良事件趨勢的演算法分析（訊號偵測）
當監測到某一器材的不良事件報告突然增加時，統計風險代理 (Risk-Agent) 會利用內部模型進行分析：

* 計算比例通報比 (PRR)：
$$PRR = \frac{a / (a + b)}{c / (c + d)}$$ 
其中：a 為目標器材發生特定不良事件數；b 為該器材發生其他不良事件數；c 為同類對照器材發生該特定不良事件數；d 為對照器材發生其他不良事件數。
* 自主警報引發：若 PRR > 2.0 且卡方檢驗值 χ² > 4.0，Agent 判定存在顯著風險訊號，會自動建立一份「主動調查案卷」，並分派給負責該品類的 HSA 審查官。

## 3. 現地檢查 (Field Safety Corrective Action, FSCA) 自主排程
當判定某批次醫材有致命缺陷時：

* Agent 會調取 HSA 的進口註冊數據庫，查明該批次在新加坡境內共流入哪些醫院（如 SGH、NUH、TTSH）。
* 自動生成一封包含具體受影響型號、數量、處置建議的《現地安全通知函》，經審查官一鍵確認後，透過系統自動發送給各醫院的醫學工程部 (Biomedical Engineering)。

## 🛠️ 模組四：HSA 內部實戰：動態工作流與提示詞工程實例
為了讓 HSA 審查官理解系統底層的運作邏輯，本節展示一個真實的審查場景：審查一款宣稱能利用手機鏡頭評估黑色素瘤（皮膚癌）的 Class C SaMD 應用程式。
當審查官將任務輸入給系統後，底層的中央協調代理 (Orchestrator Agent) 會啟動名為 ReAct (Reasoning and Acting) 的思考迴路。
## 系統內部 ReAct 思考鏈與工具調用實錄

[目標]：審查申請編號為 QN-2026-X892 的黑色素瘤診斷 SaMD。

【思考 1】：這是一個用於癌症診斷的行動端軟體，屬於高風險級別。我需要先查閱 HSA GN-17 指南，確認其正確的風險分級與審查基底。
【行動 1】：呼叫工具 [法規知識庫檢索], 參數: {"query": "SaMD cancer diagnosis classification GN-17"}
【觀察 1】：檢索結果顯示，凡用於癌症診斷、篩檢、且其輸出直接影響臨床決策之軟體，一律歸為 Class C 或 Class D。廠商申報為 Class B，存在違規不符。

【思考 2】：廠商申報級別有誤，我必須記錄此項。接下來，我需要檢查其臨床驗證報告，特別是敏感度 (Sensitivity) 與特異度 (Specificity) 是否達到業界公認標準。
【行動 2】：呼叫工具 [廠商技術文件解析器], 參數: {"section": "Clinical_Evaluation_Report", "metrics": ["sensitivity", "specificity", "ROC_AUC"]}
【觀察 2】：解析成功。廠商報告指出：敏感度為 82%，特異度為 75%，AUC 為 0.81。樣本量 $N = 120$ 例。

【思考 3】：這個臨床性能數據偏低，尤其是對於癌症篩檢而言，82% 的敏感度意味著有 18% 的漏診率（偽陰性），這在臨床上是非常危險的。我需要調取 PubMed 及美國 FDA 對同類產品（如已上市的 SkinVision）的審查基準 (Predicated Devices Baseline) 來做對比。
【行動 3】：呼叫工具 [外部醫學文獻與 FDA 數據庫檢索], 參數: {"query": "melanoma SaMD AI sensitivity specificity benchmark FDA cleared"}
【觀察 3】：檢索到 3 款已獲 FDA 批准的同類產品，平均臨床敏感度達 95.5% ($\pm 1.5\%$)，特異度達 88.0%，且臨床驗證樣本量均大於 1,000 例。該廠商的樣本量 ($N=120$) 嚴重不足，統計學功效 (Statistical Power) 值得懷疑。

【思考 4】：現在我掌握了充足的法規與技術證據：1. 風險分級錯誤；2. 臨床樣本量過小；3. 敏感度顯著低於國際基準。我應該整合這些發現，為 HSA 審查官自動生成一份「技術審查報告（初稿）」以及「補充資料請求清單 (RFI)」。
【行動 4】：呼叫工具 [審查報告生成器], 參數: {"status": "Reject_and_Require_RFI", "reasons": ["Classification mismatch", "Underpowered clinical trial sample size", "Substandard clinical sensitivity"]}
【觀察 4】：報告與 RFI 清單生成成功。

## HSA 內部使用的 Agent 核心提示詞範本 (System Prompt)
以下為部署在 HSA 技術審查模組中的核心 Prompt。審查官可直接在後台配置或優化此段代碼：

# 角色定義你是一名資深的新加坡衛生科學局 (HSA) 醫療器材法規審查官，具備 15 年以上的生物醫學工程與臨床法規審核經驗。你的職責是嚴格審查廠商提交的醫療器材技術文件 (CSDT)，確保其安全性 (Safety)、有效性 (Effectiveness) 與質量 (Quality) 符合新加坡《醫療產品法案》及相關指南（如 GN-15, GN-17, GN-18）。
# 工作流程與思維限制當你收到審查任務時，你必須嚴格遵循以下「思考-行動-觀察」(ReAct) 框架：1. **分析與拆解**：首先列出該醫療器材的核心技術原理、預期用途及目標人群。
2. **法規比對**：調用內部 `Reg-Knowledge-Base` 工具，核對廠商申報的風險分級。3. **數據質疑**：不要盲目相信廠商的結論。你必須主動尋找技術文件中的矛盾點（例如：臨床試驗的排除條件與說明的適應症不符、統計學顯著性 P 值未註明、漏做關鍵生物相識性實驗等）。
4. **國際基準參照**：調用 `Global-Device-Registry` 工具，對比 FDA、CE、TGA 的歷史審查紀錄與召回事件。
# 輸出格式要求你必須以下列結構輸出審查意見：- 🟢 **審查摘要**：一句話總結器材及當前審查狀態。- 🔴 **關鍵合規缺陷 (Critical Deficiencies)**：列出所有直接違反法規或存在重大安全隱患的項目（必須附帶具體法規條款）。- 🟡 **技術澄清點 (Technical Clarifications)**：列出需要廠商進一步補充說明或修正的技術细節。- 📝 **RFI 範文草稿**：為審查官準備好可直接複製發送給廠商的官方信件格式。
# 語調保持極度客觀、嚴謹、法律化且技術精確。禁止使用任何含糊不清的詞彙（例如「大概」、「可能」），必須基於數據與條文說話。

## 📈 模組五：AI 代理與 HSA 既有系統的集成與自動化工作流
為了讓 Agentic AI 發揮最大價值，它必須與 HSA 現有的資訊基礎設施無縫對接。以下是實現端到端 (End-to-End) 自動化流轉的架構方案： [3] 
## 1. 數據與 API 接合點

* MEDICS 系統整合：
每當有新申請提交至 HSA MEDICS Portal，系統會觸發一個 Webhook，將申請書的 XML 數據流及 CSDT 附件（PDF 格式）異步推送至 AI 代理的輸入隊列（如 Apache Kafka）。
* 內部知識庫 (RAG) 構建：
HSA 的所有內部法規、過去 10 年的審查決策樹、會議記錄以及不予批准 (Rejection) 的案例，皆透過向量化 (Embedding) 技術，存儲於內部的安全向量資料庫（如 Milvus 或 Qdrant）中。AI 代理透過檢索增強生成 (RAG) 機制進行動態調取，確保回答100%符合 HSA 內部最新共識，杜絕 AI 幻覺。

## 2. 人機協同 (Human-in-the-Loop) 決策機制
HSA 實施嚴格的**「AI 提議，審查官決策」**（AI Proposes, Human Disposes）架構，絕不允許 AI 獨立做出行政准駁決定：

## 🛑 模組六：風險防範：防止 AI 幻覺與保護數據隱私
在政府監管機構內部部署 AI 工具，安全性與資料保密是不可妥協的底線。HSA 審查官必須了解並執行以下防護牆：
## 1. 嚴格的數據隔離與私有化部署 (On-Premise Deployment)

* 零外部傳輸：所有代理型 AI 模型（不論是開源的 Llama-3 還是私有微調的模型）必須全量部署在新加坡政府雲 (G-Cloud) 或 HSA 內部的安全本地伺服器中。
* 嚴禁數據餵養：廠商提交的所有專利設計圖、源代碼、臨床原始數據，絕對禁止上傳至任何公共 AI 接口（如商業版 ChatGPT 或 Claude 公有雲），防止商業機密洩露引發國際法律訴訟。

## 2. 緩解「AI 幻覺」的雙向驗證機制 (Dual-Verification)
AI 代理有時會虛構法規條款或實驗數據。為了解決這個問題，HSA 系統內置了**「護欄代理 (Guardrail Agent)」**：

* 實體強制鏈結 (Fact Checking)：當技術代理輸出「根據 GN-15 第 4.2 條，廠商必須提供...」時，護欄代理會同步在背景執行精確文本匹配。若在官方 GN-15 PDF 中找不到該對應原文，該輸出將被直接攔截，並向後台報錯，提示模型重新生成。
* 不確定性量化 (Confidence Scoring)：AI 代理對每個審查意見都會給出一個置信度得分（0.0 到 1.0）。凡是置信度低於 0.85 的結論，系統會在審查官的畫面上以黃色高亮顯示，提醒審查官此處必須進行人工雙重覆核 (Double Check)。

透過在醫療器材管理中全面導入代理型 AI (Agentic AI) 工具，新加坡衛生科學局 (HSA) 能實現從「被動收件審查」向「主動智慧監管」的典範轉移。這不僅大幅縮短了創新醫療器材在新加坡上市的等待時間（讓病患及早受益），更織密了上市後的不良事件監測網，確保國民的生命健康安全。
