# 醫療器材來源流向追蹤預警系統規格書

## 系統名稱：TUDID 醫療器材智慧化跨國監管與預警系統 (TUDID Regulatory Guard OS)

---

## 1-1 前言

在現代全球醫療器材供應鏈中，高風險醫療器材（如國際醫療器材法規論壇 IMDRF 定義之 Class III 植入式心律調節器、人工關節及生命維持設備）的上市後監督（Post-Market Surveillance, PMS）正面臨前所未有的異質數據挑戰。跨國製造商、進口代理商、地方分銷商與醫療機構之間，數據格式高度碎片化、申報欄位缺失、全球法規代碼不一致（如 GMDN、UMDNS、WHO MeDevIS、TFDA 與 US FDA UDI 規格衝突），且缺乏即時、具備法律防禦性（Defensible）的 AI 輔助與多模型對照推論機制。

為此，本技術規格書定義了 **「TUDID 醫療器材智慧化跨國監管與預警系統 (TUDID Regulatory Guard OS)」** 的核心架構與執法官員操作指南。本系統為專門應對上述痛點而設計之高協同性、可驗證、全功能 Agentic OS 智慧化決策平台。

### 1-1-1 國際應用案例參考

本系統之設計深度參考並對標國際先進衛生主管機關之 AI 與大數據應用實踐：

* **美國 FDA (HALO / ELSA 4.0 Platform)：**
參考其利用先進自然語言處理（NLP）與機器學習技術，對不良事件報告（MAUDE 資料庫）及上市後非結構化文本進行情感分析與隱性風險挖掘的機制，建立司法證據級（Forensic-Grade）的審查蹤跡。
* **瑞士 SwissMedic：**
參考其跨國醫療器材安全警訊（Safety Alerts）動態網路拓撲與主動通報矩陣，建立跨語系、跨邊境的即時警訊解析與本地對齊網路。
* **新加坡 HSA (Health Sciences Authority)：**
參考其動態醫療器材流向追蹤與臨床替代品路徑推薦（Alternative Clinical Pathway Simulation）模型，確保在單一高風險醫材面臨全國性回收時，醫療體系之供應鏈韌性與病患安全不致崩潰。

### 1-1-2 法規與技術合規性

TUDID 系統之底層設計與資料庫架構完全相容並遵循以下國際與國內核心法規標準：

1. **US FDA 21 CFR Part 11：** 全面落實電子紀錄（Electronic Records）與電子簽章（Electronic Signatures）規範，包含非對稱加密之審計軌跡（Audit Trails）、嚴格之操作權限控管與不可篡改之系統日誌。
2. **ISO 14971 醫療器材風險管理：** 將風險鑑別、風險評估、風險控制及生產與生產後資訊導入 AI 代理（Agent）的日常推論邏輯中，以定量化風險優先指數（Risk Priority Number, RPN）作為系統動態預警之基礎。

## 1-2 現況（盤點現有資訊／資料集）

本系統的核心價值建立在對高度異質化、跨國、跨機關之資料集進行即時數據清洗、格式對齊與多維度關聯分析。以下為本系統整合之三大資料範疇：

```
+---------------------------------------------------------------------------------------+
|                                    TUDID DATA BUS                                     |
+--------------------------+----------------------------+-------------------------------+
| [SQL 結構化資料集]        | [NoSQL 非結構化資料集]      | [外部跨國網路資源 API]        |
| - 醫療器材許可證資料庫    | - 國內外醫材安全警訊文本   | - US FDA Recall Database      |
| - 單一識別系統 (UDI)      | - 醫療器材回收/下架資訊    | - SwissMedic ADR Alerts       |
| - 分類分級/來源流向申報  | - 臨床不良反應(ADR)通報單  | - TGA / Health Canada Alerts  |
+--------------------------+----------------------------+-------------------------------+

```

### 1-2-1 SQL 結構化資料

系統透過標準化 ETL（Extract, Transform, Load）管道與本地和國際結構化資料庫進行同步：

1. **醫療器材許可證資料庫：**
包含許可證字號、國內外製造廠名稱、地址、註冊效期、結構化規格描述、醫療器材標籤與說明書圖檔之元數據（Metadata）。
2. **醫療器材單一識別系統（UDI）資料庫：**
全面對接 GS1 或 HIBCC 編碼標準之 DI（Device Identifier，識別碼）與 PI（Production Identifier，生產識別碼，包含批號、序號、有效日期、製造日期），建立一物一碼之唯一對應關係。
3. **醫療器材分類分級資料庫：**
內嵌醫事字號與分類代碼對照表，兼容 IMDRF Class I/II/III 以及台灣 TFDA A至Q共十七大類之精細化分類。
4. **醫療器材來源流向申報資料：**
進口商報單、製造商出貨紀錄、盤商物流單據、醫院採購與點收紀錄，包含批號級的進銷存動態結構化數據。

### 1-2-2 NoSQL 非結構化資料

針對高自由度文字、醫院申報備註、半結構化 XML/JSON 等，系統採用分散式 NoSQL 文件資料庫進行高速檢索與語義向量化（Vectorization）儲存：

1. **醫療器材安全警訊：** 收集來自製造商自願性通報、學術期刊、醫院內部臨床工程部門申報之原廠安全修正通報（Field Safety Corrective Action, FSCA）。
2. **醫療器材回收資訊：** 包含回收原因、受影響批號範圍、處理進度（已下架、待銷毀、留置隔離）之自由文本。

### 1-2-3 其他外部跨國資源（非食品藥物管理署資料／網路資源）

系統建置跨國資料同步器（Transnational Syncer Agent），透過 API 或即時網路爬蟲（Web Scraper）定時抓取並解析以下國際權威機構之即時監管數據：

* **US FDA Recall Database：** 擷取 Class I, II, III 回收事件，自動比對國內是否有相同 DI 或同廠源之醫材。
* **SwissMedic ADR/Safety Alerts：** 監控歐洲市場最新之臨床不良反應警訊。
* **TGA (Australia) Safety Alerts：** 獲取大洋洲市場之安全通報。
* **Health Canada Field Correction Notice：** 整合美洲市場之邊境與現地修正通知。

## 1-3 使用情境（應用於醫療器材上市後管理）

本系統為 FDA 執法官員與監管決策者提供 6 個覆蓋全生命週期、高度智慧化的核心應用情境。每個情境均整合 AI 多模型對照、自動化鏈條分析與臨床替代方案推論。

### 情境一：跨國安全警訊自動化關聯與在地許可證精準對齊

* **痛點：** 美國 FDA 發布某款人工髖關節之 Class I 回收警訊，但其型號編碼、品名與台灣引進之許可證品名及規格描述完全不同，人工比對曠日費時。
* **系統動態處置：** 系統之「跨國監管代理（Global Regulatory Agent）」即時偵測到美國 FDA 回收公告，自動提取非結構化文本中的型號（Model Number）與受影響批號字串。AI 語義對齊引擎將英文描述與國內「醫療器材許可證資料庫」進行多對多相似度計算，自動識別出國內對應之許可證字號，並標記出該許可證下已進口之所有 UDI-DI 編碼。
* **執法官員檢視：** 官員打開看板，系統已自動將美國原廠警訊與本地許可證、報單數據進行 Side-by-Side 對照，並提供置信度評分（e.g., "AI 匹配置信度：98.4%"）。

### 情境二：批號級（Lot-Level）來源流向動態溯源與地理分布阻斷

* **痛點：** 某一特定批號之植入式心律調節器在臨床上發生多起電池過熱失效事件，主管機關需在數小時內查清全台所有受影響設備的具體下落。
* **系統動態處置：** 執法官員輸入該受影響批號（UDI-PI），系統立即調取「醫療器材來源流向申報資料」SQL 資料庫，運用 D3.js 動態拓撲網狀圖，在單頁視圖中繪製出該批號從進口海關、總代理、三級盤商到最終各醫療院所手術室的完整路徑圖。
* **地理空間整合：** 系統結合 GIS 地理分布地圖，將仍有庫存或已植入病患體內之醫院以紅色高亮標示，並自動計算受影響醫材之「院所留存率」與「已消耗率」，供執法官員進行精準實地稽查。

### 情境三：非結構化不良事件報告（MAUDE/ADR）之隱性風險文本挖掘

* **痛點：** 醫院端提交的不良反應報告多為自由文本，充滿臨床術語縮寫、病歷速記（如 "Pacemaker failure suspected after dynamic MRI"），傳統關鍵字檢索容易漏報。
* **系統動態處預警：** 系統之「語義解析與情感挖掘引擎」定期掃描 NoSQL 庫中的全國 ADR 通報文本。利用大型語言模型進行實體識別（NER），自動抽取出「器材組件」、「失效特徵」、「患者預後」等核心特徵向量。
* **隱性趨勢預警：** 當系統發現某特定廠牌之內視鏡在不同醫院的報告中頻繁出現「難以清洗」、「化學殘留」等不同措辭但語義相近之隱性抱怨時，系統會自動調高該產品之風險優先指數（RPN），在尚未爆發大規模集體事件前，向執法官員發出隱性趨勢預警。

### 情境四：供應鏈阻斷危機下之臨床替代路徑智慧推薦

* **痛點：** 當某款市佔率高達 80% 的人工呼吸器因重大安全缺陷必須立即全面停用時，貿然下架將導致全國加護病房（ICU）面臨設備斷炊危機。
* **系統動態處置：** 系統在官員模擬發布「全面回收指令」時，會同步啟動「臨床替代路徑模擬矩陣（Clinical Alternative Matrix）」。
* **多維度匹配推薦：** AI 代理會遍歷「醫療器材分類分級資料庫」與「醫療器材單一識別系統資料庫」，依據功能等效性（Functional Equivalency）、技術參數相容性（如通氣模式、接頭規格）及各大醫院現有之他牌庫存餘額，自動計算出最佳替代產品清單，並生成動態物流調度建議報告，確保政策執行時不會引發次生醫療災難。

### 情境五：製造商生產品質漂移（Quality Drift）多維度量化評估

* **痛點：** 醫材製造商之品質管理系統（QMS）失效通常是漸進式的，單一事件看來並不嚴重，缺乏全局量化指標導致無法及時啟動境外查廠。
* **系統動態處置：** 系統之「品質漂移監控模組」綜合加權分析該製造商在跨國（US FDA, SwissMedic, TFDA）之回收次數、臨床不良事件通報斜率、流向申報遲延率、以及歷次稽查缺陷報告（Form 483 / GMP 缺失項）。
* **動態風險畫像：** 系統為該廠商生成動態雷達圖與品質漂移趨勢線，當量化分數突破動態控制上限（UCL）時，系統自動撰寫一份結構化的「建議啟動特別稽查與境外查廠技術理由書」，交由執法官員簽核。

### 情境六：基於司法證據級（Forensic-Grade）的多模型對照稽核與合規報告自動生成

* **痛點：** AI 產出的行政處分或執法建議書，若缺乏多視角論證或存在幻覺（Hallucination），可能在行政訴訟中被廠商推翻。
* **系統動態處置：** 系統搭載「Side-by-Side 多模型司法證據級對照稽核引擎」。針對任何重大預警事件，系統同時調用多個底層語言模型（如 Google Gemini Pro、高脈絡視窗模型、專門法規微調模型）進行獨立推論與交叉驗證。
* **對照審查機制：** 系統在 Playground 視圖中並列顯示各模型的推論依據、法規條文引用準確性、以及信心指數。系統會自動比對各模型輸出的異同點，抓出潛在矛盾，最終合成一份符合 21 CFR Part 11 規範、具備完整審計軌跡與可解釋性（Explainability）的執法技術報告草案。

## 1-4 系統架構 Architecture

TUDID 系統採用全功能 Agentic OS 架構，前端基於極簡優雅的單頁大視圖設計，後端基於高效能伺服器端推論架構。

```

|  [Dashboard w/ 6 Graphs]  [Standardizer Tool]  [Skill Manager]  [Playground]          |
+---------------------------------------------------------------------------------------+
                                            |
                                            v (Express REST APIs)
+---------------------------------------------------------------------------------------+
|                                  TUDID CORE ENGINE                                    |
|  +-------------------------------------+  +----------------------------------------+  |
|  |     AI Multi-Agentic Registry       |  |  Side-by-Side Audit Inference Engine   |  |
|  | - Global Regulatory Sync Agent      |  | - Gemini Pro Regulatory Parser         |  |

                                            |
                                            v (Secure Data Access Layer)

+---------------------------------------------------------------------------------------+

```

### 1-4-2 核心層級架構說明

1. **使用者介面層（TUDID UI Layer）：**
* **Dashboard：** 整合 6 大核心視覺化圖表（D3 拓撲網狀圖、GIS 地理分布圖、風險矩陣圖、品質漂移雷達圖、替代品流動模擬圖、不良事件情感熱力圖）。
* **Standardizer Tool：** 提供直觀的異質欄位拖疊、正規化規則動態配置介面。
* **Skill Manager：** 供高級官員封裝、載入或編排特定 AI Agent 的執法技能與提示詞工作流。
* **Playground：** 多模型 Side-by-Side 司法級對照稽核與實時提示詞測試終端。


2. **核心引擎層（TUDID Core Engine）：**
* **AI Multi-Agentic Registry：** 託管多個具備自主規劃、工具調用（Tool Calling）與長期記憶能力的 AI 代理。
* **Side-by-Side Audit Inference Engine：** 負責多模型並行推論、輸出文本差異度對比（Diff Checker）、事實查核（Fact-Checking）與法規條文引用精準度驗證。


3. **數據與合規基礎設施層（Data & Compliance Infra）：**
* 負責結構化（SQL）、非結構化（NoSQL）及語義特徵（Vector Database）的混合儲存。
* 內建非對稱加密模組，對所有資料庫寫入、模型調用、官員審閱行為進行不可篡改之審計軌跡（Audit Trail）雜湊值（Hash）計算，完全滿足 21 CFR Part 11 合規要求。

## 1-5 使用需求 System Requirements Specification (SRS)

為確保系統能夠在 FDA 執法環境中高可靠度、高安全性地運行，本節詳列系統的功能與非功能性需求。

### 1-5-1 功能性需求（Functional Requirements）

#### FR-1：異質數據智慧標準化（Data Standardizing）

* 系統必須提供基於 AI 的異質數據對齊功能。當上傳不同格式的廠商申報 Excel 或跨國警訊 XML 時，AI 必須自動識別並對齊為系統標準欄位（如 `udi_di`, `udi_pi`, `manufacturer_gln`, `risk_class`）。
* 系統應支持動態規則配置，允許官員手動修正 AI 的欄位映射關係，並將該修正反饋至增量學習模型中。

#### FR-2：全鏈條流向動態追蹤與可視化（Traceability & Visualization）

* 系統必須能夠處理百萬級節點的物流流向數據。
* 必須能即時生成 D3.js 網狀拓撲圖，清晰展現「製造商 -> 進口商 -> 分銷商 -> 醫療機構 -> 病患（去識別化碼）」的完整點對點鏈條。
* 必須具備時間軸回溯功能，允許官員拉動時間滑塊，查看該批次醫材在過去任意時間點的庫存與流動分布狀態。

#### FR-3：自動化跨國監管警訊爬取與即時對齊（Global Alert Alignment）

* 系統內建的跨國監管代理必須每 30 分鐘定時輪詢 US FDA、SwissMedic、TGA 等官方資料源。
* 發現新回收事件時，必須在 5 分鐘內完成與國內許可證資料庫的語義對齊，並在 Dashboard 上將受影響的許可證標示為「國際黃色預警」狀態。

#### FR-4：21 CFR Part 11 審計軌跡與司法級報告生成（Forensic Audit & Reporting）

* 所有執法官員的操作、AI Agent 的決策路徑、多模型的原始推論文本，均必須記錄於不可篡改的稽核日誌中。
* 報告生成模組必須一鍵輸出包含：事件概述、法規引用、多模型交叉驗證結果、D3 流向拓撲圖截圖、臨床替代路徑建議之完整 PDF 報告，且該 PDF 必須內嵌基於主管機關電子憑證之數位簽章。

### 1-5-2 非功能性需求（Non-Functional Requirements）

#### NFR-1：安全性與權限控管（Security & Access Control）

* 系統必須落實基於角色的權限控管（RBAC）。普通審查員僅具備數據查詢與報告草擬權限；高級執法官員方具備核發預警、修改 Agent 技能與簽署處分書權限。
* 所有敏感數據（如病患去識別化代碼、特定醫院庫存單價）在傳輸時必須採用 TLS 1.3 加密，在資料庫內儲存時必須採用 AES-256 進行欄位級加密。

#### NFR-2：效能與回應時間（Performance & Responsiveness）

* 在處理單一產品全台流向溯源（節點數小於 10,000）時，D3 拓撲圖的渲染與生成時間不得超過 3 秒。
* 伺服器端多模型 Side-by-Side 推論的回應生成（採用 Streaming 串流技術）必須在 500ms 內吐出首字（TTFB）。

#### NFR-3：可擴展性與韌性（Scalability & Resilience）

* 系統後端微服務架構必須支援容器化彈性擴展，能夠在爆發全國性重大醫材安全事件、併發查詢流量激增 10 倍時，透過自動負載平衡確保系統不中斷。

## 1-6 特殊需求

### 1-6-1 地理資訊系統（GIS）與時空關聯分析

系統必須整合高品質之 GIS 地圖組件，不僅展示靜態點位，還需具備「時空熱力擴散模擬（Spatio-Temporal Dynamic Simulation）」能力。當某一高風險醫材在特定區域的數家醫院集中出現臨床不良反應時，系統必須能結合地理距離與物流網絡，推估是否為該區域特定低溫物流倉儲失效（如生物製劑或有溫度控制需求之骨科植入物）所導致的區域性品質裂化。

### 1-6-2 極端數據缺失下的「機率性流向推論（Probabilistic Routing）」

在上市後管理實務中，經常遇到基層盤商漏報、瞞報流向申報資料的情況，導致物流鏈條中斷。本系統之特殊需求為：當遭遇結構化數據斷鏈時，AI 代理必須能自動根據「醫院採購歷史」、「該盤商對其他類似醫材的分銷比例」、「跨國海關報單總量與國內已知銷量之差額」等多元異質特徵，運用機率圖模型（Probabilistic Graphical Models）自動推論並補全缺失的隱性流向路徑，並在拓撲圖上以虛線標示出「AI 推論隱性流向（信賴區間：95%）」，為主管機關突擊檢查提供高價值之情報線索。

### 2-2 執法官員核心操作流（Walkthrough）

為確保在真實危機發生時，官員能忙而不亂，以下梳理本系統之標準執法操作四步驟：

```
+------------------+     +------------------+     +------------------+     +------------------+
| STEP 1:          |     | STEP 2:          |     | STEP 3:          |     | STEP 4:          |
| 跨國同步與解構    | --> | 異質欄位自動對齊  | --> | 全鏈條流向阻斷    | --> | 多模型對照與處分  |
| (Global Sync)    |     | (Standardize)    |     | (D3 & GIS Trace) |     | (Forensic PDF)   |
+------------------+     +------------------+     +------------------+     +------------------+

```

1. **第一步：跨國同步與威脅解構 (Global Sync & Extraction)**
官員登入 TUDID 系統，在頂部控制列選擇當前最新的國際重大醫材缺陷事件（如：*PulseGuard-9 植入式心律調節器電池過熱故障*）。點擊 `Execute Global Sync`，系統會即時拉取 US FDA 與 SwissMedic 最新公告，利用底層語義解析器將其自動翻譯並解構為標準化的威脅特徵向量。
2. **第二步：異質申報數據自動標準化 (Schema Standardizing)**
切換至 `Data Standardizer Tool` 分頁。針對國內各大代理商提交上來的混亂報單資料（常包含不規範之欄位名稱如 `MFG_PART_NO`），點擊 `Run AI Schema Auto-Alignment`。系統將在 1 秒內自動完成與全球 UDI-DI 標準碼的對齊，並精確鎖定受影響之境內許可證。
3. **第三步：全鏈條流向精確阻斷與時空追蹤 (D3 & GIS Traceability)**
返回 `Dashboard Workspace`。此時左側的 D3 網絡拓撲圖已將該批次缺陷心律調節器從「瑞士原廠」到「台灣總代理」，再到「各區域醫學中心」的物流鏈條全線亮紅。右側的 GIS 地圖則精確標示出哪些醫院的手術室內仍有該批號庫存（黃色標記），哪些已植入病患體內（紅色閃爍）。執法官員可立即致電相關醫院採取現地隔離措施。
4. **第四步：司法級多模型對照審查與處分書簽發 (Side-by-Side Playground & Forensic PDF)**
切換至 `Side-by-Side Playground`。審查三個獨立 AI 模型針對此事件產出的法規合規性分析。比對確認各模型在引用 *21 CFR Part 11* 與 *ISO 14971* 風險控制條文時完全一致且毫無幻覺。點擊底部的 `Generate Signed Forensic Report PDF`，系統即自動打包生成一份具備完整審計追蹤軌跡、不可篡改的高質量官方執法技術報告與處分書草案。

### 3-1 跨國法規與多模型司法防禦（1-5）

2. 面對全球各主管機關對於同一款醫材之分類分級差異（例如某智慧型穿戴式生理監測設備在 US FDA 屬 Class II，但在歐盟 MDR 依新版 Rule 11 判定為 Class III），系統之「跨國監管代理」應如何建立動態權重矩陣，以避免在地許可證對齊時產生法規降級漏報之風險？
